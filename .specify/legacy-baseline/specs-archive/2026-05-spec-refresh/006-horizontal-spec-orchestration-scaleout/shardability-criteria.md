# Shardability Criteria

## Purpose

Define when feature slices can run in parallel shards versus when they must be sequenced.

## Criteria

1. Independent file ownership:
- Parallel allowed only when slices do not require writes to the same canonical files (for example constitution, templates, shared workflow manifests).

2. Shared-contract mutation:
- Sequential required when two slices both modify shared governance contracts or top-level workflow wiring.

3. Runtime dependency ordering:
- Sequential required when one slice's acceptance criteria depends on artifacts generated by another slice.

4. Extension capability readiness:
- Parallel admission requires extension preflight pass for each shard.

5. Confidence compliance:
- Shard step execution is autonomous only at confidence >= 80%.
- If confidence < 80%, that shard must pause for human input while other compliant shards may continue.

## Conflict Triggers

- Same file write intent in shard candidates.
- Shared release gate ownership in same execution window.
- Explicit dependency edge from candidate A to B.

## Deterministic Sharding Rule

- Sort candidates lexicographically by feature directory.
- Place non-conflicting candidates into parallel shard lanes in sorted order.
- Place conflicting candidates into a deterministic sequential lane in sorted order.
